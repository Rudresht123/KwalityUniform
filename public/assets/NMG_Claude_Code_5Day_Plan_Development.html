Day 1

Foundations & Setup

Goal: Everyone has Claude Code installed, authenticated, and has run a first agentic session, and understands the 3-tier model cost strategy and how to monitor spend. Covers guide sections 1, 2, 3 and 15. Claude Code is NOT a chatbot — it is an agent that acts on your code.

#TaskTypeLearning Resource
1.1Understand what Claude Code is — an agentic CLI that reads your codebase, runs commands, edits files and manages git. It is not a chatbot; it acts on your code (§1)Read▶Overview — Watch↗Official Docs — Overview
1.2Install Claude Code on your OS — macOS (curl or brew), Windows native (winget) or WSL2, or Linux (apt/dnf/apk). Native installer ships a self-contained binary; no Node/Python required (§2)Setup▶Installation — Watch↗Official Docs — Setup
1.3Authenticate with claude auth login (or ANTHROPIC_API_KEY), then run claude in a real repo and ask it to read & summarise the codebase (§2)Setup↗Official Docs — Quickstart↗Claude Code 101 course
1.4Learn the 3-tier cost strategy: Haiku for simple/fast tasks (saves 60–70%), Sonnet for ~80% of daily work, Opus for deep reasoning only. Pick a default (see the model table below) (§3)Read▶Choosing the right model↗Docs — Choosing a model
1.5Practice switching models mid-session: /model claude-sonnet-4-6, /model claude-haiku-4-5, /model claude-opus-4-8 (§3)Build↗Docs — Model config
1.6Set up token & credit monitoring: /usage for the 5-hour window and claude credits status for balance & burn rate. Note programmatic vs interactive pools (table below) (§15)Check↗Docs — Costs↗Manage usage credits

Best practices — new projects (§7)
StepAction
1Start with CLAUDE.md — define stack, conventions, forbidden actions
2Run /init — Claude reads codebase and writes CLAUDE.md
3Set up MCP servers early — connect GitHub, DB, issue tracker from day 1
4Write first skill for any repeated workflow (e.g. deploy checklist)
5Default to Sonnet, escalate to Opus only when needed
6Use /batch for parallel feature development across worktrees
Best practices — existing projects (§8)
StepAction
1Run /init first — Claude analyses codebase and creates CLAUDE.md
2Add .claude/rules/*.md with path-specific rules (frontend vs API)
3Start with read-only tasks — ask Claude to explain codebase before editing
4Use Opus for first deep-dive into unfamiliar code, then switch to Sonnet
5Add hooks for safety — run tests automatically before Claude marks task done
6Use subagents for isolated research — won't pollute main context

Reference · Session restore (§17)
# Core commands claude --continue # resume last session (fastest) claude -c # shorthand claude --resume # interactive session picker claude -r # shorthand claude --resume abc123 # resume specific session by ID claude --session-name "x" # start with a named session # Team naming convention (standard) claude --session-name "feature-user-auth" claude --session-name "bugfix-payment-null" claude --session-name "refactor-api-layer" claude --session-name "qa-checkout-flow"
Context window strategy (§17)
SituationAction
Session reaching 60% context/compact — Claude summarizes and continues
Switching to a different task/clear — fresh start, edits stay
Resuming next day, same featureclaude --continue — full history restored
Handing off to a teammatePush CLAUDE.md to git + share session ID
Memory types (§17)
TypeLocationStoresSurvives /clear?

Reference · Team session workflow (§17)
Dev A: claude --session-name "feature-checkout-v2" Works on feature → /compact at 50% → pushes CLAUDE.md to git Dev B: git pull claude -c # (or claude -r to pick session from picker) → Full context restored, continues seamlessly
Known limitations (§17)
IssueWorkaround
--continue in -p mode may silently create a new sessionAlways verify session ID in scripts
Context not always fully restored on --resumeRun /recap after resume to verify state
Remote session ends if terminal closesUse tmux or screen to keep terminal alive

Reference · Laravel CLAUDE.md template (§20)
# Laravel Project Standards ## Stack: PHP 8.3 | Laravel 11 | MySQL 8 | Redis | PEST ## Architecture - Controllers: thin only — no business logic - Business logic: app/Services/ - Database access: app/Repositories/ — never Eloquent in controllers - API responses: always via app/Http/Resources/ - Validation: Form Requests only — no inline $request->validate() ## For Every New Endpoint 1. Route in routes/api.php 2. Form Request class 3. Service method 4. API Resource 5. PEST feature test ## Commands - php artisan serve | php artisan migrate - php artisan test | php artisan queue:work

MySQL best practices (§20)
TaskKey rule
Schema designAsk Claude to normalise — then review before applying
Migration safetyAlways provide row count — batch operations for tables > 500k rows
Query optimisationAlways paste EXPLAIN ANALYZE output — without it Claude is guessing
Rollback firstAsk for rollback migration before the forward migration
